## Acceptance gate + determinism (ROADMAP v2, Step 1)

The eval is now a **gate**, not just a report. `python evals/run_eval.py --gate <label>`
re-runs and compares to `evals/<label>.json`; a regression in any **hard** metric (graph
correctness — `impact_fp/fn`, `wrong_symbol_same_name`, `duplicate_card`, `ambiguous_ref`;
or retrieval quality — `precision@1/3/5`, `mrr`, `resolution_rate`) exits non-zero, so M3's
acceptance rule ("ships only if it doesn't regress error or balloon tokens") is enforced
mechanically and M1's caller-graph can't regress unnoticed. Token metrics warn, not block.

Making that gate trustworthy required fixing a real **non-determinism**: `hybrid_search`
sorted tied combined-scores in `all_ids` **set-iteration order** (PYTHONHASHSEED-dependent),
so two runs over the *same* index could flip a gold file between rank 2/3 — moving
`precision@3` by a full query (0.067) with no code change. A deterministic chunk_id
tiebreak (`sort(key=lambda x: (-score, chunk_id))`) makes ranking reproducible across
processes; `--perquery` is now byte-identical run-to-run. A baseline can only be moved by
re-saving it — a deliberate act, never silent drift.

**Re-baseline 2026-06-20 (session 3, deliberate, user-authorized).** After Step 5 + the C++
fixture, `index --full` was re-run and `roadmap_v2_baseline` re-saved, so Step-6 mode weights
calibrate against the current repo rather than a stale index. Two corpus/gold corrections
came first: (1) `evals/fixtures/` is now excluded from the main index (the C++ corpus is
indexed only by `--cpp`), so fixture C++ symbols don't pollute the Python gold; (2)
`IMPACT_GOLD` for `is_test_path` was re-derived from an exhaustive grep — **6** real callers
(it had a stale **3**, omitting the pre-existing `graph._split_prod_test`, `find_tests`, the
`test_graph` unit test, and predating `brief._partition_tests`) — so `impact_fp/fn` stay 0 by
**truth**, not by matching the tool. The ranking **algorithm is unchanged**, so the retrieval
moves are pure dogfood drift on the larger corpus: `precision@1` 0.600→0.467 and `mrr`
0.667→0.584 (new distractors — a grown `evals/gold.py`, the new `brief.py`; `brief.py` is a
top-1 for none of the 15), while `precision@3` 0.600→0.667 and `resolution_rate` 0.667→0.733
improved. Graph-correctness metrics stayed 0/0. **Run `--gate` and `--cpp` together** going

**Caller-graph matrix (M1 + #11 graph dimension):** `tests/test_caller_graph_regression.py`

## Search trust primitive (ROADMAP v2, Step 2)

`retrieval/confidence.py` + `Retriever.search_assessed`: a result set is judged LOW
confidence when **the top hits collapse onto one bare symbol name while a query domain term
is covered by none of them** — the measured T2 `"cell size"` → `.size()` failure. On low
confidence the retriever fans out per-term sub-queries and re-ranks the union by domain-term
coverage (the manual T2 retry, automated — no LLM, no ban-list), and `repo_search` prints a
one-line trust banner. The exact-symbol fast path is high-confidence and never fans out.

**Honest measurement (M3):** on the 15 Python gold queries the tightened detector fires on
**1/15** (down from a too-eager 9/15 before the name-cluster requirement) and the eval gate
is **non-regressive** — `precision@1/3/5`, `mrr`, `resolution_rate` all unchanged vs
baseline, tokens slightly lower. That is the *correct* result, not a null one: the failure
Step 2 targets is a **bge-embedding** collapse of a compound query (`cellSize`/`CellGrid` →
`size`) on a **C++** corpus; the Python gold + bag-of-words fake embedder can't reproduce a
buried-but-recoverable target, so the detector rightly stays quiet here. The recovery
mechanics (clustering detection, coverage re-rank promoting the buried domain term, the
fan-out gate) are locked by `tests/test_confidence.py`. Step 2 is therefore **gate-verified
non-regressive and unit-verified**, and — as of the C++ fixture below — **now also measured
against its real target.**

### C++ retrieval fixture — the win, finally measured (`run_eval.py --cpp`)

`evals/fixtures/cpp_grid/` is a tiny C++ corpus that reproduces the exact T2 failure under
the **real bge model** (the Python gold + FakeEmbedder can't): eight domain-neutral `.size()`
accessors plus one true target, `Grid::rescaleCellSize`, whose doc says "dimensions" not
"size" so the embedding never ranks it on the `size` token. `run_eval.py --cpp` indexes it
fresh with the real model and asserts categorical correctness (not a drifting metric). It is
a **runnable** regression check, not a standing lock — it needs the real model, so it can't
live in pytest or `--gate`; **run it alongside `--gate`** at the end of each step:

| query | true target | base rank | final rank | fan-out | verdict |
|---|---|---|---|---|---|
| `cell size` | `Grid::rescaleCellSize` | **6** (buried under the `.size()` cluster) | **0** | fired | recovered |
| `ring buffer size` (control) | `RingBuffer` | 0 | 0 | none | no over-fire |

So the detector fires on the real collapse and the coverage re-rank promotes the buried
target **rank 6 → 0**, while a genuine container-size query stays high-confidence and is
unmoved. This is the win Step 2 was built for, no longer taken on faith.

**Scope of this measurement (honest).** It covers the **atomic-term** compound query
(`cell size` → terms `{cell, size}`), which `rerank_by_coverage` recovers because those terms
are already split. It does **NOT** cover the **single-compound-token** case (`cellSize` →
term `{cellSize}`): there the tokenizer-asymmetry residual below means `assess`/`rerank` see
coverage 0 and can't recover it. That residual is the documented Step-2 follow-on; this same
fixture is where a `cellSize`-style case would measure it once the shared tokenizer is
addressed (a deliberate, gate-affecting change, not made here).

**Tokenizer-asymmetry residual (found in the Step-5 review, Step-2-rooted).** `assess` builds
its query terms from `query_terms` (which keeps a compound identifier like `update_transform`
WHOLE) but compares against `result_terms` (which SPLITS to `update`,`transform`). So a
compound query term is always in `assess.missing_terms`; when it also trips a name-cluster,
`assess` → low and any consumer (the search banner, and `repo_brief`, which withholds before
its own coverage check runs) treats a recoverable query as low-confidence. `rerank_by_coverage`
shares the asymmetry, so fanout won't reliably recover it either. This bites **C++ hardest**
(CamelCase/snake/qualified identifiers) — the same blind spot as the buried-C++-target caveat
above, and it would be measured by a `cellSize`-style case in the **C++ fixture now built**
(see "C++ retrieval fixture" above; the atomic `cell size` case there already passes).
Deliberately NOT patched
here: `assess`/`rerank_by_coverage` are gate-locked and tuned, and the brief must not
out-guess the shared detector (it would disagree with the `repo_search` banner). `repo_brief`'s
own `_coverage` IS fixed (splits its qset), so the residual only fires on the compound-term
**and** name-cluster conjunction, not on ordinary compound-term queries.

### Scope of these metrics (honest)

The gold set is **100% Python files of this repo**, so every number above measures the
**Python** system. They do **NOT** exercise the multi-language (C++/C#/JS/TS) graph edges
shipped alongside them. Multi-language edge correctness — including the cross-language
**non**-resolution guarantee (a C++ `helper()` must never resolve to a Python `helper`) —
is validated separately by unit tests in `tests/test_multilang.py`, not by these eval
numbers. Don't read the impact FP/FN here as evidence the C++ graph is accurate; read the
multilang tests for that.

## repo_brief — capstone synthesis (ROADMAP v2, Step 5)

`brief.py` (`repo_brief` + `render_brief`) composes the shipped trust primitives into one
pre-flight entrypoint: a **HEURISTIC** block (task interpretation, ranked likely targets,
likely call flow — guesses from Step-2 search) **hard-separated** from a **VERIFIED** block
(impact, confident callers, related tests, staleness, risks — graph facts about the one
anchored target via `edit_plan`). The behavioral fork is binary: at LOW anchor-confidence
the verified block is **withheld** (the brief degrades to candidates + a disambiguation
action), because a confident-wrong brief is worse than no brief.

**A/B vs `context_pack`** (`python evals/run_eval.py --brief`, live index, 15 gold queries).
This is a **separate** mode — it never touches `run()`'s summary, so the `--gate` baseline
is unmoved (gate stayed PASS, all hard metrics flat).

| metric | value | reading |
|---|---|---|
| brief_top3_hit_rate | 0.600 | **==** retrieval precision@3 — the likely-targets ARE the search hits; the brief adds no retrieval magic and claims none |
| pack_file_hit_rate | 0.933 | apples-to-oranges depth: the pack lists ~10 files, the brief's top-3 is 3 |
| anchored_rate | 0.800 | committed on 12/15; **withheld on the 3 low-confidence cases** (1 a would-be-wrong anchor correctly suppressed; see next row) |
| anchor_precision_when_committed | 0.583 | of the 12 commits, 7 hit the gold file (≈ symbol precision@1 — the anchor is the top symbol hit) |
| withheld_would_have_hit | 2/3 | of 3 withholds, 1 was a would-be-wrong anchor correctly suppressed; 2 were correct-but-low-confidence |
| avg_brief_tokens | 848.3 | **cheaper** than a pack (no code bodies — refs + graph only) |
| avg_pack_tokens | 1716.1 | — |

`confidence_dist {high:2, medium:10, low:3}` on this run skews to MEDIUM because the dogfood
index is **stale relative to in-session edits** — the Step-5 staleness cap correctly demotes

**Honest verdict (M3).** A brand-new synthesis tool can't be scored by `run()` (which never
calls it), and the static gold set **cannot** measure the real M3 win — fewer wrong edits /
re-searches across a whole task. So the brief is accepted on what *is* measurable: it is
**gate-non-regressive**, **cheaper than the context pack** it replaces as the
edit-from-intent starter (848 vs 1716 tokens — see the table above), and
**integrity-preserving by construction** —
the withhold gate makes a confident-wrong verified block structurally impossible. The
conservatism is real and reported: the gate suppressed 2 anchors that would have hit, to
guarantee it never anchors on a low-confidence guess (those targets are NOT lost — they
stay in the heuristic block with a fetch order). The live smoke is the canonical case: for
*"how does query rewrite and confidence gating work"* retrieval surfaced `embed_query`

The task-level token + error win is stated as a **hypothesis** the static harness can't
verify — the same honesty pattern as the Step-2 note.

## Context modes — weight presets (ROADMAP v2, Step 6)

`retrieval.modes` ships three RANKING-WEIGHT presets (scope untouched — Phase 4 settled
symbol-only), selected per call via `mode=` on `repo_search`/`repo_context_pack` (CLI
`--mode`); absent/unknown → the tuned default. Presets are PRINCIPLED, not grid-searched:
`impact` {sym .55 / kw .25 / vec .20} (favour the exact symbol you'll edit), `discovery`
{sym .25 / kw .25 / vec .50} (favour semantic recall for vague intent), `bugfix`
{sym .25 / kw .55 / vec .20} (favour literal keyword/error tokens).

**Why they ship UNVALIDATED.** A mode is proven only by DIFFERENTIAL (crossover) performance
— preset A beats default on type-A queries WHILE default beats preset A on type-B. That needs
≥2 well-populated query types (the #11 matrix), which doesn't exist; the gold is 100%
discovery-shaped. Grid-searching a handful of self-authored queries while judging on them is
circular (n=6 ⇒ one query = 0.17, noise). So presets are first-principles + labelled
hypotheses — the Step-2/Step-5 honesty pattern.

**What the discovery-only gold CAN show** (`run_eval.py --modes`, fresh index):

| mode | P@1 | P@3 | P@5 | MRR | paired vs default (n=15) |
|---|---|---|---|---|---|
| default | 0.467 | 0.667 | 0.800 | 0.584 | — |
| impact | 0.400 | 0.600 | 0.667 | 0.500 | 0 better, 6 worse (OFF-TARGET) |
| **discovery** | **0.533** | **0.867** | **0.933** | **0.706** | **7 better, 0 worse** |
| bugfix | 0.133 | 0.600 | 0.800 | 0.391 | 1 better, 6 worse (OFF-TARGET) |

- **`discovery` is the only ON-TARGET row** (the gold IS discovery) and beats the default
  on every metric (P@3 +0.20, P@5 +0.13, MRR +0.12, 7/0 paired) — a single-type signal, on a
  **self-authored** discovery gold, that a vector-heavy weighting helps discovery retrieval
  here. Suggestive, not independent.
- impact/bugfix are OFF-TARGET here (their query types aren't in the gold), so being worse on
  discovery queries is EXPECTED, not a failure — these presets are simply tuned away from
  semantic recall. This does **not** demonstrate query-type *sensitivity*: single-type
  degradation is equally consistent with "the discovery weighting is just globally better."
  The **crossover** that actually justifies separate modes (a preset winning on its OWN query
  type while losing on others) cannot be shown from one query type — it needs the #11 matrix.

**Two honest non-claims.** (1) This does NOT validate impact/bugfix — they need #11.
(2) The default is deliberately NOT retuned toward discovery: tuning the global default on the
only measurable query type would overfit (it could hurt the impact/exact-symbol queries this
gold can't measure — the same crossover gap). So `discovery` is offered as a mode for
discovery tasks while the default stays put. `bugfix`'s premise (distinctive error/log
literals) is additionally THIN on this Python repo (grep finds ~none), so it is the least
exercised — validate it on a repo that has them.

## Step 8 — C++ header→cpp declaration-doc merge (2026-06-20)

**The gap.** A C++ method is routinely DECLARED with its Doxygen/`///` doc in a header but
DEFINED out-of-line in a `.cpp` (`void World::queueDeath() {...}`) where the definition has no
comment. The parser only emits the definition, so its embedded descriptor collapses to the
bare signature and the header's doc — usually the only prose — never reaches the vector/keyword
channels. This is the C++-priority payoff #5 (doc capture) could not realize across the
header/cpp boundary.

**What shipped.** `tree_sitter_parser.cpp_decl_docs` mines a header's function *declarations*
for their doc comments, keyed by a canonical qualified name (`namespace/class stack + leaf`,
`::`→`.`) that matches the `.cpp` definition's `qualified_name` after the same normalization.
At index time `index_runner` reads a translation unit's sibling header(s) (same-dir stem +
`src/`↔`include/` mirror) and, when a definition has **no local doc**, uses the header doc as
the summarizer's `preceding` (one doc-extraction path) and records a `decl_ref` (header
declaration site) on the symbol + chunk, surfaced by `repo_get`. Gated by
`indexing.cpp_header_merge` (default on). **Declarations are never emitted as symbols** — only
their docs travel — so `by_name`/`by_qname`/`resolve_call` are provably unchanged (the
IMPROVEMENTS.md "Bigger bets" hard constraint). Per the user's call, the spec's "alternate
decl-site" is delivered now as `Symbol.decl_ref`, not deferred.

**Measured (`run_eval.py --cppmerge`, real bge model).** Fixture `evals/fixtures/
cpp_header_merge/` isolates the merge: the query words (`defer / destruction / frame /
complete`) appear ONLY in the header doc — never in the target's name/signature/`.cpp`, and the
header is `.pandemoniumignore`d so the **only indexed carrier** of those words is the merged
definition's summary; decoy methods (`completeFrame`, `flushDestructionQueue`, …) carry the
words in their *names* to bury the target without the merge.

| query | rank OFF | rank ON | verdict |
|---|---|---|---|
| "defer an entity's destruction until the frame is complete" → `World::queueDeath` | **2** | **0** | PASS |

The buried out-of-line definition is recovered to rank 0 once its header doc is merged. Like
`--cpp`, this needs the real model so it is a RUNNABLE categorical check (run alongside
`--gate`/`--cpp` at step end), not a pytest lock. The deterministic mechanism — the doc lands
on the definition's stored summary, `decl_ref` is stamped, gated by config, across same-dir and
`src/`↔`include/` layouts and both out-of-line styles (`namespace {…} World::m` and
`sim::World::m`) — is locked offline in `tests/test_cpp_header_merge.py` (+14 tests,
suite 132→146 including the nested-class + overload-suppression cases the review added).

**No-op everywhere else (verified).** `--gate roadmap_v2_baseline` after a full reindex: all
HARD metrics flat (Δ +0.000); the lone move is soft `avg_pack_tokens` +24.9, which is
Step-6/7 source↔index drift now that the index reflects current source — **not** this change
(which never touches Python or the header-less `cpp_grid` fixture). `--cpp` PASS unchanged.

**Known limitation — SILENT cross-file staleness (documented, not solved).** The merge is
realized at index time and is self-contained per `.cpp` (it reads the header from disk). If
**only** the header's doc comment changes, the `.cpp` is not reindexed, so its merged summary
**and its `decl_ref`** (a point-in-time `header:line` pointer carried verbatim, never
re-resolved) go stale **and the staleness detector will not flag it** — staleness is anchored
to the `.cpp`'s own content hash, which did not change. This differs from #1/#5 (a one-time reindex to
activate): it is an ongoing silent gap. Closing it (a header→cpp dependency reindex) is scope
the roadmap demotes; for now, a full reindex re-merges. A header→cpp edit on the SAME `.cpp`
is fine — the `.cpp` is reindexed and re-reads the current header.

## #9 — Channel-isolation baselines + external task sets (2026-06-22)

**The gap this closes.** The harness compared the tool to its OWN prior snapshot (`--gate`)
and across scopes/models (`--sweep`/bake-off), but **never to single-channel retrievers** —
the "0/4 retrieval baselines" gap. So we could say "we didn't regress" but not "hybrid earns
its complexity over plain vector / plain BM25 / plain symbol."

**What shipped.**
- **`run_eval.py --baselines`**: runs the gold set under **hybrid vs symbol-only vs
  keyword-only(BM25) vs vector-only**. Isolation is a new additive `channels_only` arg on
  `Retriever.search`/`search_assessed`/`_base_search` (`hybrid_search.py`) — it runs ONLY the
  named channels, so the merge ranks by that channel alone (absent channels contribute 0; the
  `_normalize`/`_identity` maps are monotonic → faithful single-channel ranking, **not**
  weight-zeroing). Symbol-only / keyword-only never touch `self.lance`/`self.embedder`, so
  they pay no model load. It bypasses the exact-symbol short-circuit and the fan-out (both are
