My honest answer: **your map is good enough as a first-pass navigation layer, but not yet good enough to reliably make an LLM infer all correct relations by itself.**

It is strong for:

```text
Where is this symbol?
What calls this?
What does this call?
What imports this?
What inherits from this?
Where are nearby tests?
What files changed?
What is a likely architectural area?
```

It is weaker for:

```text
What business workflow depends on this?
What config/data contract silently affects this?
What event/queue/route triggers this?
What UI behavior depends on this backend field?
What mutation causes this later failure?
What relation exists only through framework convention or dependency injection?
```

So the design should not be:

```text
Give LLM repo_map and hope it figures out relations.
```

It should be:

```text
Give LLM a typed, evidence-backed relation graph + exact refs + confidence labels.
```

Your current architecture is already pointing in that direction: `repo_search` returns cards, `repo_get` fetches exact code, and the graph answers related/affected questions. The graph stores static `calls`, `imports`, and `inherits` edges unresolved at index time, then resolves them at query time with receiver-aware rules like `self.m()` and `mod.m()`. That is a good precision-first design. 

## Verdict

I would grade the current map like this:

| Capability                           |         Current quality | Why                                                                                                                         |
| ------------------------------------ | ----------------------: | --------------------------------------------------------------------------------------------------------------------------- |
| Repository orientation               |                **Good** | `repo_map` modes such as architecture, entrypoints, domains, tests, and changed give the agent a useful top-level picture.  |
| Symbol navigation                    |                **Good** | Cards, qualified refs, summaries, tags, and exact `repo_get` are a strong retrieval base.                                   |
| Static call/import/inherit relations | **Good but incomplete** | Query-time resolution is the right design, but static extraction will miss dynamic/framework relations.                     |
| Test relation discovery              |            **Moderate** | The tools expose tests, but test relevance often needs naming, imports, fixtures, mocks, and call-path heuristics.          |
| Business/domain relation discovery   |    **Weak to moderate** | `repo_logic_map` and domains help, but the `affects` layer is dormant, so semantic coupling is not first-class yet.         |
| Edit-impact prediction               |            **Moderate** | `repo_impact` can help through callers/files/tests, but without data/config/event edges it will miss many real impacts.     |
| Relation correctness guarantees      |      **Not enough yet** | You need evidence, confidence, ambiguity reporting, and evals before trusting it as “correct.”                              |

The key point: **your map is good enough to guide the LLM, but not enough to be the LLM’s only source of relational truth.**

## Why the current approach is directionally right

Modern repo-level coding research is moving toward graph-aware retrieval because plain similarity retrieval often finds code that looks related, not code that is structurally related. RepoGraph explicitly argues that file-level indexing can find semantically similar files while missing genuinely related snippets, and proposes repository-level graph navigation to expose dependencies and interactions. ([arXiv][1])

CodexGraph follows the same direction: it integrates LLM agents with graph database interfaces so agents can make structure-aware, multi-hop code queries instead of only searching text. ([ACL Antoloji][2])

GRACE also reflects the same trend: repository-level tasks need structural relationships such as call chains and inheritance hierarchies, not just text similarity; it builds multi-level code graphs with file structures, ASTs, calls, class hierarchies, and data-flow semantics. ([arXiv][3])

So your instinct is correct: **a map is necessary.** The question is whether your map has enough relation types, evidence, and confidence labels.

Right now, it is a strong **static relation skeleton**, not yet a full **semantic project map**.

## The biggest risk

The biggest risk is that an LLM will see a plausible map and infer too much.

For example, if the map says:

```text
OrderService.approve -> EmailService.send
```

the LLM might assume:

```text
Changing approve affects all email templates.
Changing send affects order approval behavior.
The missing vendor email bug must be in EmailService.
```

Some of that may be true. Some may be false. The map needs to tell the agent:

```text
relation_type: static_call
confidence: verified
evidence: services/order.py:144 calls email_service.send(...)
direction: approve calls send
scope: direct
not_proven: whether send is always reached
not_proven: whether this is vendor email specifically
```

Without that, the LLM may treat a weak relation as a causal fact.

## What your map currently does well

### 1. It separates search from exact code

This is excellent. The architecture says cards are cheap references with summary/tags, while exact code is fetched by `repo_get(ref)` and remains the source of truth. 

That prevents the map from becoming a hallucinated replacement for code.

### 2. It uses static graph edges for hard relations

Calls, imports, and inheritance are the right base graph. These are the relations an LLM should not have to guess. Your graph stores unresolved edges and resolves them at query time, which avoids stale cross-file edges after partial reindexing. 

That is a strong architectural decision.

### 3. It is conservative

The architecture says the graph prioritizes precision over recall because a wrong “X affects Y” edge misleads the agent. 

That is exactly the right bias for agentic coding. Missing an edge is annoying. A false edge can cause a bad edit.

### 4. It already exposes the right tools

The README lists `repo_graph`, `repo_impact`, `repo_edit_plan`, `repo_logic_map`, `repo_find_tests`, `repo_brief`, and `repo_context_pack`. That is the right surface area: the LLM should be able to ask not only “where is X?” but also “what touches X?” and “what should I inspect before editing X?” 

## Where the map is not enough yet

### 1. Static call edges do not capture business causality

A call graph can tell you:

```text
A calls B
```

It cannot always tell you:

```text
A depends on B’s side effect
A must happen before B
A and B share a data contract
A writes a value later read by B
A publishes an event consumed by B
A changes a database row that affects B
```

Those are the relations that usually matter in real bugs.

Your architecture acknowledges this with the `affects` layer, but that capability is dormant: there is read-path support, but no normal CLI/MCP/skill workflow that generates and ingests those edges. 

So today, the map can show **structural relations**, but not enough **semantic coupling relations**.

### 2. Framework conventions will be missed

Many important relations are not ordinary function calls:

```text
HTTP route -> controller/action
controller -> service through dependency injection
model -> serializer/schema
migration -> ORM model
template -> view/controller
React component -> prop contract
Flutter widget -> route/state provider
event producer -> event consumer
queue/topic -> handler
config key -> runtime behavior
environment variable -> feature branch
```

The README says static graph edges are extracted for Python, C++, C#, Dart, and JS/TS, while HTML/CSS are parsed into symbol anchors but carry no edges; other languages may be text-indexed with explicit notices. 

That is reasonable for an MVP, but it means the LLM still needs help connecting framework-level relations.

### 3. `repo_map` is probably too coarse by itself

A map that says:

```text
services/
controllers/
models/
tests/
```

helps orientation, but it does not prove relation correctness.

A good relation map needs to answer:

```text
Why is this related?
What kind of relation is it?
Is it verified or inferred?
Where is the evidence?
What exact ref should I fetch?
What uncertainty remains?
```

Without those, an LLM will fill gaps with naming guesses.

### 4. Similarity is not relation

Your architecture includes “similar implementations” via vector neighbors. That is useful, but it should be labelled very clearly as similarity, not dependency. 

For example:

```text
PaymentService.refund
InvoiceService.void
```

may be similar because they both reverse transactions. But neither may affect the other.

The map should visually and textually separate:

```text
verified dependency
probable dependency
semantic similarity
same domain
same test cluster
LLM hypothesis
```

## What I would add

### 1. Add a relation confidence ladder

Every edge should have a class like this:

```text
verified_static
  Extracted from syntax. Example: direct call, import, inheritance.

verified_framework
  Extracted by a framework-specific rule. Example: FastAPI route decorator,
  ASP.NET controller action, Express router registration.

heuristic
  Based on naming/path/test conventions. Example: tests/test_order.py likely tests
  services/order.py.

semantic
  Based on descriptor/vector similarity. Useful, but not causal.

llm_hypothesis
  LLM-inferred relation with evidence. Never treated as fact.

unknown_or_ambiguous
  Name matched multiple candidates or resolution failed.
```

Then the LLM can reason safely.

Do not let everything appear as simply:

```text
related
```

That word is too vague.

### 2. Add evidence to every relation

A relation should be returned like this:

```json
{
  "from": "services/orders.py::OrderService.approve",
  "to": "services/email.py::EmailService.send_vendor_email",
  "type": "calls",
  "confidence": "verified_static",
  "evidence_ref": "services/orders.py:142-146",
  "evidence": "self.email.send_vendor_email(order)",
  "resolution": "receiver-aware self.email resolved from constructor assignment",
  "direction": "from_calls_to",
  "depth": 1
}
```

For hypotheses:

```json
{
  "from": "config/email.yaml::vendor_po_template",
  "to": "services/email.py::build_vendor_email",
  "type": "config_affects_behavior",
  "confidence": "llm_hypothesis",
  "evidence_refs": [
    "config/email.yaml:12-18",
    "services/email.py:55-72"
  ],
  "why": "Function reads template key used by vendor purchase-order email.",
  "needs_verification": true
}
```

This is the difference between a map that **suggests** and a map that **supports reasoning**.

### 3. Add “relation packs,” not only maps

For an LLM, a raw graph can still be too abstract. I would add a relation-oriented context output:

```text
repo_relation_pack(ref, task, budget)
```

Output:

```text
Target:
  services/orders.py::OrderService.approve

Verified direct relations:
  calls -> EmailService.send_vendor_email
    evidence: services/orders.py:144
  calls -> OrderRepository.save
    evidence: services/orders.py:150
  imported_by -> controllers/order_controller.py
    evidence: controllers/order_controller.py:9

Likely test relations:
  tests/test_order_approval.py::test_vendor_email_sent
    reason: imports OrderService and calls approve

Possible semantic relations:
  config/email_templates.yaml::vendor_po_template
    reason: same tag vendor-email + config key used by callee
    confidence: heuristic

Not resolved:
  self.event_bus.publish("order.approved")
    no local consumer found
```

That is much more useful than a generic repo tree.

### 4. Add entrypoint-to-symbol flows

For feature work and bugs, the LLM usually needs a path like:

```text
HTTP route / CLI command / event handler / UI action
  -> controller
  -> service
  -> domain method
  -> repository
  -> external side effect
  -> tests
```

So `repo_logic_map(topic)` should not only return relevant symbols. It should return **flows**:

```text
Flow: approve purchase order
1. routes/purchase_orders.py::POST /purchase-orders/{id}/approve
2. services/purchase_order.py::PurchaseOrderService.approve
3. domain/purchase_order.py::PurchaseOrder.approve
4. repositories/purchase_order.py::save
5. services/email.py::send_vendor_po_email
6. tests/test_purchase_order_email.py::test_vendor_email_sent_on_approval
```

Each edge should show whether it is verified static, framework-derived, heuristic, or semantic.

### 5. Add data/config/event edges

This is probably the biggest missing upgrade.

I would add edge types like:

```text
reads_config
writes_config
uses_env
reads_table
writes_table
uses_model
serializes_with
validates_with
publishes_event
subscribes_event
uses_queue
uses_topic
uses_template
uses_feature_flag
renders_component
passes_prop
```

These relations are often more useful than call edges for real-world bugs.

Example:

```text
OrderService.approve
  publishes_event -> "purchase_order.approved"

VendorEmailHandler.handle
  subscribes_event -> "purchase_order.approved"

Therefore:
  approve affects vendor email indirectly
```

A normal call graph may completely miss that.

### 6. Make ambiguity explicit

When resolving names, do not silently choose a plausible target.

Return:

```text
Ambiguous:
  call "send" could resolve to:
    - EmailService.send
    - NotificationClient.send
    - QueuePublisher.send

No edge emitted because receiver type was unknown.
```

This helps the LLM avoid false confidence.

Your architecture already says precision over recall, which is right. The next step is to expose unresolved/ambiguous edges as useful warnings, not just omit them. 

### 7. Give the LLM “what to inspect next”

A map should be prescriptive.

For any target ref, return:

```text
Before editing this symbol, inspect:
1. Direct callers
2. Direct callees with side effects
3. Interface/base class
4. Nearest tests
5. Config/schema/event contracts
6. Similar implementations
7. Recently changed related files
```

This makes the map agent-operational, not just informational.

## The relation types I would support

I would design the relation schema around these groups:

```text
Structural code relations:
  contains
  member_of
  calls
  called_by
  imports
  imported_by
  inherits
  implemented_by
  overrides
  instantiates

Framework relations:
  route_to_handler
  handler_to_service
  dependency_injected
  signal_to_handler
  command_to_handler
  component_uses_component

Data/config relations:
  reads_config
  uses_env
  reads_table
  writes_table
  uses_model
  uses_schema
  serializes_with
  validates_with
  uses_template

Event/async relations:
  publishes_event
  subscribes_event
  sends_queue_message
  consumes_queue_message
  schedules_job
  job_handler

Test relations:
  tested_by
  fixture_for
  mocks
  asserts_behavior_of

Semantic relations:
  similar_to
  same_domain
  same_workflow
  possible_affects
```

Then every relation should carry:

```text
source_ref
target_ref
relation_type
direction
confidence
evidence_refs
extractor
depth
is_transitive
is_runtime_possible
is_verified
notes
```

## What “good enough” should mean

I would not define map quality by whether it looks nice. I would define it by whether it helps an agent avoid wrong edits.

A good map should pass tests like these:

### Bug task

```text
Task: “Vendor email is not sent after purchase order approval.”
```

Map should surface:

```text
approval entrypoint
approval service/method
email sending callee or event
vendor email template/config
tests around approval/email
recent changed files, if any
unresolved event consumers, if any
```

### Refactor task

```text
Task: “Rename ApprovalStatus.APPROVED.”
```

Map should surface:

```text
enum definition
all references
serializers/API schemas
database migrations/seeds
tests
UI labels or TypeScript types
docs/config if indexed
```

### Behavior change task

```text
Task: “Approval should require manager role.”
```

Map should surface:

```text
authorization middleware
approval endpoint
service method
user/role model
tests/fixtures
similar role checks elsewhere
```

If the map cannot produce those relation packs, the LLM will still need to rediscover too much.

## Recommended implementation priority

I would upgrade in this order:

1. **Keep the current static graph.** Calls/imports/inherits with query-time resolution are the right foundation. 
2. **Add relation metadata:** confidence, evidence refs, extractor, ambiguity, direction.
3. **Make `repo_logic_map` return typed flows, not just relevant symbols.**
4. **Make `repo_context_pack` graph-aware by default.** It should include target code, parent context, direct callers/callees, tests, and config/event/schema relations when available.
5. **Add framework extractors for your target stacks.** Do not try every framework first; support the repos you actually care about.
6. **Wire up the dormant `affects` workflow.** But label it as hypothesis, not fact. 
7. **Add relation evals.** Build 20–50 real tasks and manually mark expected related refs. Track precision@k, recall@k, and “agent found correct edit site.”

## My final answer

Your map is **good enough for navigation**, and your static graph is a **solid base for relation discovery**.

But it is **not yet good enough to make the LLM reliably figure out correct relations alone**, especially for dynamic, framework-driven, config-driven, event-driven, or business-domain relationships.

The design goal should be:

```text
repo_map gives orientation.
repo_graph gives verified structural facts.
repo_logic_map gives typed workflow flows.
repo_impact gives change consequences.
repo_context_pack gives minimal complete evidence.
```

The LLM should not be asked to infer relation truth from a pretty map. It should receive **typed relations with evidence and confidence**, then use exact code to verify before editing.
