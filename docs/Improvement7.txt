The Claude-Code-specific truth that reframes everything else: **your competition isn't other RAG systems, it's ripgrep + Read, and Claude Code reaches for those by reflex.** I grep by default because grep is always loaded, always free, and I'm trained to death on it. So the question that decides whether Pandemonium gets used is never "is the retrieval good" — it's "for *this* query, do I reach for repo_search or just grep, and does anything make that choice for me?" If the skill doesn't make that boundary a reflex, I default to grep and your index sits cold. Everything below is downstream of that. Ranked by Claude-Code-side leverage:

**1. Make the grep-vs-search boundary a reflex in the skill, not a paragraph.** The decision rule has to be crisp enough that I apply it without deliberating: grep for things I already know the name of — exact strings, an identifier I'm tracking, mechanical find-replace targets, anything where I can spell the token. repo_search for the queries grep *structurally cannot answer*: "where is X handled" when I don't know the symbol, "what's related to this," "what breaks if I touch this." The failure mode is subtle — for a query where grep would *also* work, I'll take grep every time because it's already in hand, even when search would've been better. So the skill shouldn't say "search is available for semantic queries"; it should say "if you're about to grep for a *concept* rather than a *token*, stop and repo_search instead." Frame it as interception of my default, because that default is strong.

**2. Your 15 MCP tools are a permanent context tax — paid every turn, every session, before any work happens.** This is the most underweighted Claude-Code cost in your design. MCP tool schemas sit resident in my context the entire session; 15 tools with descriptions and params is plausibly 2–5k tokens of overhead I'm carrying on turn 1 and turn 200 alike, whether or not I use them. Native Grep/Read cost me nothing because they're built in. So you're starting every session already behind on the budget you're trying to save. Collapse the surface: a few fat tools beat fifteen thin ones, and the rarely-used ones (`repo_logic_map`, `repo_edit_plan`, `repo_impact`) can live behind the *skill* — taught as procedures the skill invokes — rather than as always-loaded MCP schemas. The tension is real (more tools = more discoverable), but discoverability you pay for once in the skill; schema tax you pay every turn.

**3. Compaction is your single unique opening — the ledger is memory that outlives my context window, which native Claude Code does not have.** When my context fills, Claude Code auto-compacts: it summarizes the conversation and *drops the detail* — including everything I retrieved and reasoned about. That's the moment I'm most likely to re-search what I already found and re-fetch what I already read, because the evidence is gone and only a summary remains. `repo_session` is the fix native Claude Code can't offer: if it persists and is queryable *after* a compaction, I can re-hydrate — "what refs did I fetch, what facts did I confirm" — instead of rediscovering. But this only works if the skill explicitly tells me to consult the ledger after losing context, because post-compact I won't *know* I had a prior session unless something reminds me. This is the highest-value Claude-Code-specific capability you have and it's currently passive. Make it the thing the skill reaches for on re-orientation.

**4. Generate the CLAUDE.md skeleton at index time — that's where "the resident map" lives on this platform.** The architectural skeleton I said I need maps exactly onto CLAUDE.md, because CLAUDE.md is auto-loaded every session and never evicted. So don't make me *call a tool* to orient — have `pandemonium index` emit a compact module map (top neighborhoods, their roles, where requests enter, dependency direction) as a block the user drops into CLAUDE.md, or write it directly. Then orientation is free and always-on instead of a tool round-trip I have to remember to make. This is the cheapest high-leverage artifact you can ship, and it's Claude-Code-native: you're feeding the one context channel that's guaranteed resident.

**5. Shape the heavy retrieval around the subagent boundary.** Claude Code subagents (the Task tool) each get their *own* context window — which is the cleanest token lever available to you, and you already use subagents for `affects` generation so you have the pattern. The play: a noisy investigation — five searches, two graph walks, three dead leads — should run *inside a subagent*, which burns its own context and hands the parent back only distilled refs + findings. The parent stays clean. `repo_context_pack(task, budget)` is already exactly this shape; the missing piece is the skill teaching me to *delegate* big retrievals rather than running them in my main context and polluting it with the dead leads. "If a retrieval will take more than ~3 searches, spawn a subagent for it" is the kind of rule that changes my token profile dramatically on a 1000-file repo.

**6. repo_get's only durable edge over native Read is symbol re-resolution — so make me fetch by symbol, not by line.** I can already Read any file with an offset and limit, so line-range fetching gives me no reason to leave the native tool. The one thing Read *can't* do is survive edits: in a multi-edit session, line numbers drift, and a ref I grabbed 20 turns ago as `file:340-365` now points at the wrong code — silently. `path::Symbol`, re-resolved live by name, doesn't drift. That's the concrete, defensible reason to route through repo_get, and the skill should sell exactly that: "fetch by symbol ref so it stays valid as you edit." Lead with edit-stability, because line-fetching alone, I'll just use Read.

**7. RAG faithfulness hole specific to your design: descriptor staleness.** You embed descriptors and deliver code — which means I match on the *summary* and act on the *code*, and those are two different artifacts that can drift apart. After I edit a function, its code changed but its descriptor (and the vector built from it) still describes the old behavior. `repo_changed` tells me the *file* changed, but does the descriptor regenerate, or do I keep retrieving that symbol on a summary that now lies? In a long editing session this compounds: the index increasingly describes a codebase that no longer exists, and I retrieve on promises the code stopped keeping. This is the RAG correctness issue your architecture is uniquely exposed to *because* you decoupled embed-target from delivery-target (which is otherwise the right call). At minimum the skill should warn me that descriptors for files I've edited this session may be stale and the code is ground truth; better, mark cards whose underlying file changed post-index as "summary may be stale."

Two closing observations, one a strength worth naming and one a quiet mismatch:

The strength: **your card model is already LLM-reranking, and that's the right architecture for me specifically.** Classic RAG leans on a reranker because the embedding recall set is mediocre. You hand me cards — ref + summary + tags, no code, cheap — and let me pick before fetching. That *is* a rerank pass, with me as the reranker, and it's better than a model-based reranker because I'm conditioning on the actual task, not a generic relevance score. The implication you should exploit: your recall set can be *looser* than a no-rerank RAG would tolerate, because I'll prune it on the cards for nearly free. Wider net, cheap LLM filter — that's a strong design and you should tune toward it rather than away.

The mismatch: **I query lexically even when your vector channel wants concepts.** My instinct, shaped by years of grep, is to search for identifier-shaped tokens — so I'll feed your conceptual vector channel queries like `getUserById` when it'd shine on "user lookup by identifier." Your hybrid merge absorbs some of this, but I'll chronically under-feed the channel that's most differentiated from grep. If the skill nudged query formulation — "describe what the code *does*, not what you think it's *named*" — you'd get more out of the one channel that justifies the whole vector index. Small skill change, disproportionate effect on whether the semantic half earns its keep.
