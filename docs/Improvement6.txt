Before the list: three of these (A, B, C) have impact that's entirely conditional on your corpus, and one cheap measurement decides how much they're worth — so I've ordered by impact-per-effort and flagged the conditionality inline. And since your own key-decisions table shows intuition has misled here before (file/code scope *hurt* precision in the bake-off), anything that changes the retrieval distribution should ride the same eval harness rather than going in on argument. I'll say which items need that and which don't.

**Tier 1 — correctness, ship without an eval**

**Reconcile the config default (`index_runner.py:255` vs `settings.py`).** Logically this is two sources of truth disagreeing on one default; make the fallback `["symbol"]`. There's no performance *gain* here — it's a performance *protection*: it stops a programmatic caller who omits the scopes key from silently getting `["symbol","file","code"]`, which is 3× the chunks plus the two scopes your evals say degrade precision. True impact is narrow — only the no-key config path, i.e. embedding integrations and tests that drift into prod — but the failure is silent and precision-degrading, which is the worst combination. Token effect is indirect (more chunks → noisier retrieval → more marginal cards delivered), and intelligence effect is indirect (the hurt-precision scopes mislead ranking). Hygiene, one line, no measurement needed.

**Include decorators/attributes in the symbol span.** Take tree-sitter's `decorated_definition`, not the bare `function_definition`; same for C# `[HttpGet]`/`[Authorize]` attribute lists. Pure correctness, zero performance cost — same parse, outer node. Where it bites: framework code, where the decorator *is* the most identifying fact (`@app.route("/users/<id>")`, `[HttpPost("{id}")]`, `@pytest.fixture`, `@property`). For your .NET MVC controllers this is high; for the C++ sim it's near-zero (no decorators). Token cost is trivial — a line or two — but the signal-per-token is the highest in this whole list, because "where's the DELETE /users endpoint" is answered by the decorator and not the body. Dropping it blinds the tool to routing and test wiring; that's the intelligence argument. No eval needed — this strictly adds correct information to a span.

**Tier 2 — the real retrieval-quality change**

**Synthesize a "module body" card for top-level code.** This is the #1 gap from before, and it's the one that changes results on *your* corpus specifically, because controllers, `Startup.cs`/DI registration, route tables, and CLI/argparse blocks are exactly the app-wiring code you work in daily — and in symbol-only mode it's covered by nothing (file has symbols → not a no-symbol file → no 60-line windows; file emitted chunks → backstop never fires). Logic: compute the complement of covered line ranges within the file, drop import-only and whitespace/comment-only runs (imports are already edges), and if a contiguous residue run exceeds a floor, emit one card spanning it with qname `path::<module>`, enrichment-eligible.

The objection writes itself — *didn't the bake-off show `code` scope hurts?* — and the answer is why this is safe where that wasn't: `code` windows hurt because they laid arbitrary 60-line slices *over already-symbol-covered code*, producing redundant low-coherence cards that competed with the clean symbol cards and diluted ranking. The module-body card covers the *complement only*, so it never overlaps a symbol and it's a coherent unit ("module wiring"), not an arbitrary slice. The failure mode of `code` structurally can't occur here. (Bonus: macro-defined or codegen'd functions that tree-sitter misses fall into this residue and become findable instead of vanishing.)

Performance is negligible — one extra descriptor per file that has uncovered top-level lines. The token argument is the strong one and it's net-*saving*: today an agent asked "where are routes registered?" either fails or falls back to `repo_get(expand=file)` and eats the whole file; the module-body card lets it find the wiring block directly and fetch only that. So it removes whole-file fetches, which are your most expensive delivery path. Intelligence-wise it enables a question class the tool currently can't answer at all ("what runs at import," "where's the app configured"). This one I *would* A/B against baseline — not because I doubt the coverage logic, but to confirm the new cards don't crowd slots on library-heavy corpora where the residue is thin.

**Tier 3 — token economy (the core thesis)**

**Put a size hint on the card, and make `repo_get` default safe.** Add `loc` + an `est_tokens` estimate to the card metadata — free at index time, you already own the span. The point isn't the number, it's the default it should drive: `repo_get(expand=exact)` on a symbol over some token threshold should return `head:N` plus a "truncated, M more lines, view=full to expand" notice rather than dumping the whole thing unbid. Logic: right now the agent picks a ref blind to cost and a single minified-JS or generated symbol is a multi-thousand-token context bomb with no warning, and the `view=signature|head:N` controls only help if the agent already *knows* the symbol is huge — which it can't, pre-fetch. Performance: free. True impact is occasional but sharp — minified/generated JS-TS (relevant to your Flutter/web), vendored single-file libs, god-functions; near-zero on a corpus of small functions. This is the most direct lever on the tool's entire reason to exist: it's the difference between a 400-token fetch and a 4,000-token one, and a safe default means the tool protects the agent's budget even when the skill forgets to. The intelligence framing is that it turns a blind fetch into a cost-aware one — exactly the "manage the agent's attention" thesis, applied to the agent's own decisions.

**Tier 4 — decide, don't keep half-built**

**`ast_block` children: give them distinct descriptors or delete the layer.** This stays conditional on the answer to my earlier question, and the first move is a measurement, not code: log the fire rate — what fraction of symbols exceed 60 lines and produce ≥2 blocks. If it's under ~10%, cAST barely runs and the honest call is to drop it until it earns its place; don't carry half-built machinery. If it's meaningful, then the logic is that a child chunk exists *only* as a retrieval target (delivery upgrades to the parent), so a child whose vector is the parent's descriptor is a literal no-op that adds index bloat and zero precision — four near-identical vectors all collapsing back to one parent. Distinct descriptors fix that: a 200-line parse→validate→transform→persist function becomes matchable at the validate sub-region, so "where do we validate the payload" ranks it for the right reason and still delivers the whole coherent symbol. Performance cost is real but bounded — N extra enrichment summaries per large symbol, plus the child vectors — and concentrates on god-functions, which is where navigation help matters most. Token delivery is unchanged (still the parent); the win is landing on the *correct* parent instead of a wasted fetch + re-search, and the parent-upgrade rule already guards against a child polluting results against its own parent. Modest intelligence gain, narrowly targeted at your worst-structured code. Measure first.

**Tier 5 — strategic ceiling, different lane**

**Wire a minimal `affects` produce→ingest, or stop rendering the read paths.** Out of the chunking lane, but it's the highest-ceiling item and your own doc flags it as dormant: ingest API exists, nothing produces shards, so no edges ever exist and the sections in `repo_graph`/`repo_edit_plan`/`repo_brief` always render empty. The logic problem is that an always-empty section is worse than an absent one — an agent reading the skill may treat "no affects edges" as "nothing affects this," a false-negative trap on the exact question (what breaks if I change this constant?) that static call graphs can't answer. True impact if it works is large — cross-file config/contract/ordering coupling is precisely what an editing agent needs — but it's LLM-hypothesis, so it carries the highest misleading-risk, which is why your precision-over-recall graph philosophy gated it. Token-wise empty sections waste a little and populated ones add high-value tokens. Intelligence ceiling is the highest in the whole system; so is the precision risk. So this is a real project with confidence-gating, not a fix — but in the meantime, guard the read paths to render nothing rather than an empty "Affects:" header that an agent can misread.

**Sequencing and the one thing to instrument first**

Do the config reconcile and the decorator span today — correctness, no eval. Then add two log lines on your next indexing run before writing any more code: a histogram of the covered-line *complement size* per file (tells you A's payoff on your actual corpus) and a histogram of *symbol line count* (tells you B's fire rate). Those two numbers convert A and B from arguments into decisions. Then land the module-body card behind an A/B, then the size hint, then resolve B from the fire-rate data. Treat `affects` as its own track.

The honest one-liner: if your indexed corpus is mostly application/service code (which yours is), A is the highest-value item here and C is close behind. If it were mostly the C++ sim, both would be near-zero and the size hint (D) would lead instead — the sim is where unbounded single-symbol fetches are most likely.

Honest answer, and it cuts against the grain of your own roadmap: **find is the part you've mostly already solved.** What actually limits me in a 1000-file repo isn't locating code — the card model handles that — it's *changing code safely when I can't run it*. My expensive failures are write-side, not read-side. So "what would help me" pulls toward closing the write→verify loop, not sharpening retrieval further. Ranked by how much each would actually change my outcomes:

**1. Impact *completeness*, not impact precision — invert the gate for the impact path specifically.** Your precision-over-recall choice is correct for ranking and wrong for "what will I break." The asymmetry flips between the two: in a ranked search list a missed caller costs me a little; in impact analysis a missed caller is a broken build I commit *confidently* and never see. What I need from `repo_impact` is not a clean list of 6 resolved callers — it's "6 resolved, **plus 3 unresolved call sites with this name I could not attribute — go verify.**" Tell me your recall gap out loud. A tidy list that's silently short is the single most dangerous output you can hand me, because it's the one that makes me break things while feeling certain. This is the highest-value change in the list.

**2. A breakage check *after* an edit.** I write code and can't run it. Most of my catastrophic mistakes aren't "couldn't find it" — they're "found it, made a plausible edit, was wrong in a way a compiler catches in 200 ms." You already own the symbol graph, so you can answer "did this edit break name resolution" without a full compile: which refs no longer resolve, which import is now dangling, which signature no longer matches its N call sites. That post-edit check would prevent more bad outcomes than any retrieval improvement you could ship. It's the highest-leverage thing you *haven't* built.

**3. A cheap architectural skeleton I keep resident all session.** When a human joins a big repo they get the tour — layers, entry points, dependency direction. I get dropped at a random file. A card tells me what a symbol *is*, never where it *sits*. Give me a budget-bounded (~1–2k token) map of neighborhoods — top modules and their roles, where requests enter, which way dependencies point — small enough to never get evicted. Then every card I fetch lands somewhere on a map instead of in a void, and I stop making the "right edit in the wrong layer" mistake. `repo_map`/`repo_logic_map` gesture at this; the binding requirement is *small and stable enough to stay resident*, not comprehensive.

**4. Make `repo_get` re-fetch-aware through the ledger.** The card model killed whole-file reads; the silent killer underneath is *re-reading*. I re-fetch symbols I've already seen — because I don't remember I saw them, or I saw `head:N` and now need full. The ledger already tracks fetched refs, so `repo_get` should know it: "you saw this at full in turn 12, unchanged," or better, return *only the diff* since I last saw it. Diff-only re-fetch is real token savings on long sessions and nearly free given what the ledger already stores.

**5. Deterministic outputs — trust is load-bearing in a way that's easy to underrate.** If the same query returns different ordering across calls, I burn context reconciling it, and worse, I stop trusting the tool. The failure mode is specific: the moment I *half*-trust you, I do **both** — use the tool *and* read the files myself to check — which is strictly worse than either alone. A tool I fully trust changes my behavior; a tool I half-trust doubles my work. Reproducibility isn't polish, it's the thing that decides whether I actually offload to you or quietly route around you.

**6. Task-conditioned card annotation, thin layer.** My real waste isn't search, it's the *fetch on a dead lead* — that's what pollutes context. Your summaries are computed at index time, so they're task-agnostic by construction. A thin task-time pass that annotates *why this card matters for this task* (or just re-ranks the result set against the task) lets me prune harder and commit to fewer fetches. `repo_context_pack(task, budget)` is already the right shape; the open question is whether ordinary single-card search results get any task conditioning or none.

**7. Treat `affects` as "go look," never as fact — and notice it's the grounded edges I actually act on.** As the consumer: an LLM-inferred "X affects Y" that I then *act on* is a pipeline for laundering a hallucination straight into an edit. Your labeling (confidence + evidence, clearly marked) is exactly right and I'd want it loud. But for me, grounded edges plus a verify loop beat speculative edges every time — because I can *act* on a grounded edge and can only *investigate* a speculative one. Which is why I'd put #2 far ahead of wiring `affects` into a workflow.

The meta-point, and the thing I'd actually want you to take away: every item above is really the same request wearing different clothes — **tell me the boundaries of what I know.** Tell me your recall gap (#1), tell me what my edit just broke (#2), tell me where I am in the system (#3), tell me what I've already seen (#4). The one thing I'm genuinely worst at generating on my own is *calibration* — knowing when I don't know. I can produce plausible code and plausible confidence with equal ease, and in a 1000-file repo the gap between those two is exactly where things break silently. A retrieval tool feeds me information. The tool that would actually make me a better engineer in a large codebase feeds me **the edges of my information** — because that's the part I can't supply myself, and it's the part that determines whether my next edit is correct or just confident.
